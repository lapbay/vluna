# Metadata Override Mechanisms (OSS)

This document summarizes the metadata-driven override mechanisms that are used
when Vluna materializes grant assignments and ledger grants. It also clarifies
which entities participate, the metadata fields involved, and which downstream
fields are overridden.

## Summary (Important)
- Not all metadata-driven behavior in the codebase uses `normalizeGrantBindingOverride`.
- The override parser is used specifically for **grant issuance** flows, where
  metadata contains `grants` or `grant_override`.

Core parser: `normalizeGrantBindingOverride`  
Location: `vluna/oss/packages/vluna-core/src/services/grant-issuance.service.ts`

## Common Override Parser: normalizeGrantBindingOverride

The override parser accepts a single object that describes a grant program and
optional override fields. It is used in multiple sources (see below). The
following fields are recognized (canonical names shown; several camelCase aliases
also accepted in code):

Required:
- `grant_program_code` (aliases: `program_code`, `programCode`)

Optional override fields:
- `amount_xusd` (alias: `amountXusdOverride`) -> overrides ledger grant amount
- `scale_by_invoice_quantity` (alias: `scaleByInvoiceQuantity`)
- `on_ledger_override` (alias: `onLedgerOverride`)
- `issue_anchor_override` (alias: `issueAnchorOverride`)
- `window_kind_override` (alias: `windowKindOverride`)
- `window_relative_seconds_override` (alias: `windowRelativeSecondsOverride`)
- `alloc_seq_override` (alias: `allocSeqOverride`)
- `priority_override` (alias: `priorityOverride`)
- `kind_override` (alias: `kindOverride`)
- `metadata` (object merged into ledger_grants.metadata)

### Downstream fields affected
When these overrides are applied, they affect the following fields on
`ledger_grants`:
- `amount_xusd` (with optional quantity scaling)
- `on_ledger`
- `period_start` / `period_end` (via `issue_anchor_override`)
- `window_start` / `window_end` (via `window_kind_override` + `window_relative_seconds_override`)
- `alloc_seq`
- `priority`
- `kind`
- `metadata`

Reference: `issueGrantForAssignment` in
`vluna/oss/packages/vluna-core/src/services/grant-issuance.service.ts`.

## Metadata Sources That Use the Override Parser

### 1) Catalog product + price metadata (checkout/subscription grants)
Origin:
- Catalog price mapping metadata (e.g. Stripe price mapping) plus optional product metadata.
- Used during purchase/checkout processing.

Metadata path:
- `catalog_products.metadata.grants` (array or object)
- `catalog_prices.metadata.grants` (array or object; overrides product-level entries when
  the same `grant_program_code` appears)

Materialization:
- Creates `grant_assignments` and immediately issues `ledger_grants`.
- Location: `processGrants` in
  `vluna/oss/packages/vluna-core/src/services/billing-purchase.service.ts`

Notes:
- Overrides are used to determine amount, window, anchor, and ledger posting.
- Quantity is derived from the provider session/subscription; `scale_by_invoice_quantity`
  controls whether the quantity multiplies `amount_xusd`.

### 2) Billing plan metadata (plan-driven grants)
Origin:
- `billing_plans.metadata.grants` (array or object).

Metadata path:
- Each entry is parsed via `normalizeGrantBindingOverride`.
- Optional non-override fields supported in plan grants:
  - `effect` (allow/deny)
  - `template_key`
  - `template_meta`

Materialization:
- Creates/maintains `grant_assignments` for active billing plan assignments.
- Stored assignment metadata includes:
  - `grant_template` (key/hash/meta)
  - `grants` (array with the normalized override)
- Location: `ensureBillingPlanGrantsEnrollmentSynced` in
  `vluna/oss/packages/vluna-core/src/services/billing-plan.service.ts`

Override application:
- `issueGrantsForAccount` reads `grant_override` or `grants` from the
  assignment metadata to apply overrides during `ledger_grants` issuance.

### 3) Grant campaign metadata (ops campaigns)
Origin:
- `grant_campaigns.metadata.grants` (array or object).

Metadata path:
- Each binding element is parsed via `normalizeGrantBindingOverride`.
- Binding window fields are read from the *binding record itself*:
  - `binding_window_start` / `binding_window_end`
  - (also accepts `bindingWindowStart`, `bindingWindowEnd`, `window_start`, `window_end`)

Materialization:
- Campaign sweep creates `grant_assignments` for target accounts.
- Assignment metadata includes:
  - `grant_override` (single normalized override object)
  - `grant_campaign_id`
- Location: `grant-binding-sweep.task.ts` in
  `vluna/oss/packages/vluna-core/src/tasks/grant-binding-sweep.task.ts`

Override application:
- The assignment-level `grant_override` is applied when issuing `ledger_grants`.

### 4) Grant assignment metadata (override carrier)
Origin:
- Generated by billing plan or campaign workflows.

Metadata path:
- `grant_assignments.metadata.grant_override` (single override object), or
- `grant_assignments.metadata.grants` (array of overrides).

Materialization:
- Used directly during grant issuance.
- Location: `issueGrantsForAccount` in
  `vluna/oss/packages/vluna-core/src/services/billing-plan.service.ts`
  and grant binding sweeps that issue grants.

## Fields That Are NOT Part of normalizeGrantBindingOverride

These are commonly used alongside overrides but are *not* parsed by the override
normalizer:
- `binding_window_start` / `binding_window_end`
  - Used only in campaign binding records.
  - Control the `grant_assignments.window_start/window_end` when created from a campaign.
- `template_key`, `template_meta`, `effect`
  - Used in billing plan grants to form template metadata and planning logic.

## Quick Reference: Source -> Metadata -> Target

- Catalog product + price mapping metadata
  (`catalog_products.metadata.grants` + `catalog_prices.metadata.grants`)
  -> Grant assignments + immediate ledger grants
  -> Overrides apply to `ledger_grants` fields.

- Billing plan metadata (`billing_plans.metadata.grants`)
  -> Grant assignments (source_kind=billing_plan_assignment)
  -> Overrides stored in assignment metadata and applied to `ledger_grants`.

- Grant campaign metadata (`grant_campaigns.metadata.grants`)
  -> Grant assignments (source_kind=ops.campaign)
  -> Overrides stored as `grant_override` in assignment metadata.

- Grant assignment metadata (`grant_assignments.metadata.grant_override` / `grants`)
  -> Applied during issuance of `ledger_grants`.
